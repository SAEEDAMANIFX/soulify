# Soulify (سابقاً SmartRig Pro) — وثيقة تصميم: نظام الملابس + الوجه + المقتنيات + الأنيميشن
**الإصدار:** مسودة 2 — 2026-07-02 — للنقاش، لا شيء نهائي

---

## 1. الرؤية

Soulify يتحول من "أوتو-ريقر للجسم" إلى **منصة كاملة: ريقينغ + أنيميشن** صالحة لإنتاج أفلام سينمائية في بلندر:

```
Soulify
├── تبويب RIG (نظام الخطوات الحالي، يتوسع)
│   ├── Body Rig        (موجود — الماركرز + Go! + Match to Rig)
│   ├── Face System     (وحدات وجه معيارية، أساسه كورس Storm)
│   ├── Garment System  (وحدات ملابس معيارية)
│   ├── Props System    (حقيبة، سيف، عصا... مع Parent Switching)
│   └── Deform Toolkit  (شيب كيز / درايفرز / كوركتيف)
└── تبويب ANIMATE (يظهر بعد اكتمال الريق)
    ├── جهاز القيادة (Locomotion Controller)
    ├── باكات الحركة (قتال، أنماط مشي...)
    ├── Lipsync من الصوت
    ├── مكتبة البوزات والتعابير
    └── التكيف مع الأرضية (Ground Adaptation)
```

**الدمج مع نظام الخطوات:** نظام الخطوات الحالي يبقى العمود الفقري. كل وحدة جديدة = خطوة اختيارية بعد اكتمال الجسم. تبويب ANIMATE يُقفل حتى يكتمل `Match to Rig` — يضمن ترتيباً منطقياً ويمنع الأخطاء.

**المبدأ الحاكم:** لا ريق منفصل لكل قطعة. كل قطع الملابس تُختزل إلى **5 وحدات عامة** والفرق بينها بارامترات وإعدادات مسبقة (Presets).

## 2. البنية المعيارية الموحدة (Rig Module)

كل وحدة (ملابس أو وجه) تلتزم بواجهة واحدة:

| المرحلة | الوظيفة |
|---------|---------|
| `detect()` | تحليل هندسي: هل هذا الميش يناسب الوحدة؟ + اقتراح البارامترات |
| `fit()` | توليد عظام مرجعية مطابقة للميش (نفس فلسفة SR_Reference) |
| `build()` | بناء عظام التحكم والقيود والدرايفرز |
| `skin()` | الأوزان (تلقائي أو نقل من الجسم) |
| `controls()` | خصائص الـ UI الخاصة بالوحدة (سلايدرات، مفاتيح) |

**فائدة الالتزام بالواجهة:** إضافة قطعة جديدة مستقبلاً = وحدة جديدة أو Preset، بدون لمس النواة. ونفس البنية تخدم وحدات الوجه.

## 3. وحدات الملابس الخمس

### 3.1 وحدة الأسطوانة (Tube/Skirt) — موجودة، تحتاج تعميم
**تغطي:** تنورة قصيرة، تنورة طويلة، كندورة، فستان قصير/طويل، عباءة.

الموجود حالياً: fit على edge loops الرأسية (`_skirt_grid_topo`) + fallback زاوي، rest coords، jig/tweak/STRETCH_TO.

**التعميم المطلوب:**
- **صفوف متعددة** (rows): التنورة القصيرة صف تحكم واحد؛ الكندورة/الفستان الطويل 3–5 صفوف حسب الطول (نسبة الطول إلى الورك تحدد العدد تلقائياً).
- **Follow Legs** (اختياري): عظام الجهة الأمامية تتبع الفخذ بنسبة قابلة للضبط (Copy Rotation بتأثير جزئي) — أساسي للكندورة عند المشي.
- **No-Intersect** (اختياري): Shrinkwap بوضع Outside ضد ميش الجسم على عظام tweak مع offset — يمنع دخول القماش في الساق.
- **قسم علوي للفستان:** الفستان = جزء علوي fitted (وحدة 3.3) + تنورة (هذه الوحدة). الأداة تكتشف خط الخصر وتقسم تلقائياً.
- **Presets:** Short Skirt / Long Skirt / Kandura / Dress / Abaya — كل Preset = قيم جاهزة للبارامترات نفسها.

### 3.2 وحدة الحبل المتدلي (Cord/Dangle)
**تغطي:** ربطة عنق، Lanyard، Whistle cord، حاملة البطاقة، قلادة، وشاح، حزام متدلٍ.

**التصميم:**
- سلسلة عظام على المحور الطولي للميش (استخراج medial line — نفس فكرة `_finger_trace` لكن لشريط).
- نقطة تثبيت (Anchor) بـ Child-Of على عظمة الجسم (رقبة/صدر/حزام). ربطة العنق: تثبيت عند العقدة + عظمة مستقلة للعقدة نفسها.
- الحلقة حول الرقبة (Lanyard): نصفان مثبتان على الرقبة + الجزء المتدلي حر.
- **No-Intersect:** Shrinkwap (Outside) ضد الصدر على عظام السلسلة + Limit Distance من عظمة الصدر. مفتاح تشغيل/إيقاف + سلايدر offset في الـ UI. هذا هو الحل العملي — لا محاكاة تصادم فعلية (غير مستقرة ومكلفة).
- **Dynamics اختياري:** إما baked wiggle بسيط (درايفر تأخير على الدوران) أو توصية المستخدم بـ Wiggle 2 — يُناقش.
- **Presets:** Tie / Lanyard / Whistle Cord / ID Badge / Necklace.
- البطاقة/الصافرة في نهاية الحبل = عظمة rigid واحدة (وحدة 3.4) مربوطة بآخر عظمة في السلسلة.

### 3.3 وحدة الملابس الملتصقة (Fitted / Attach to Body)
**تغطي:** قميص، بنطال، جاكيت عادي، جوارب، قفازات.

**ليست ريقاً — أداة "Attach to Body" بزر واحد:**
1. Data Transfer للأوزان من ميش الجسم (nearest surface interpolated).
2. تنظيف: قصّ الأوزان الضعيفة + Limit Total + Smooth.
3. ربط بنفس الأرميتشر.
4. **كشف تلقائي للعظام المؤثرة فقط** (الجاكيت لا يحتاج أوزان أصابع القدم).
5. اختياري: عظام تصحيحية جاهزة للياقة، الأكمام، حافة الجاكيت السفلية (تُبنى كـ tweak bones صغيرة يحركها المستخدم عند الحاجة).

**ملاحظة مهمة:** جودة النقل تعتمد على تطابق الميشين. نضيف فحص "فجوة المسافة" — إذا كانت القطعة بعيدة عن الجسم (جاكيت واسع) ننقل عبر **Proxy mesh** (نسخة منفوخة من الجسم) لنتائج أنظف.

### 3.4 وحدة الإكسسوار الصلب (Rigid/Socket)
**تغطي:** قبعة، نظارة، حذاء، نعال، ساعة، حزام صلب.

- عظمة واحدة (أو Child-Of مباشر) على socket: الرأس للقبعة، القدم للحذاء.
- كشف تلقائي للـ socket الصحيح: أقرب عظمة جسم إلى مركز حجم القطعة.
- **الحذاء حالة خاصة:** يحتاج أوزان foot + toe (وليس عظمة واحدة) + يفتح موضوع **Foot Roll** — أصلاً في قائمة العمل المفتوح للجسم، والحذاء يجعلها أولوية.
- النعال: نفس الحذاء + خيار "heel free" (الكعب ينفصل عن القدم عند رفعها) — عظمة نعل مستقلة بـ Copy Location على مقدمة القدم فقط.

### 3.5 وحدة السحاب (Zipper) — الأصعب والأكثر تميزاً
**تغطي:** جاكيت بسحاب، سترة هودي، حقيبة.

**التصميم المقترح (يحتاج نموذج تجريبي قبل الالتزام):**
- المستخدم يحدد حافتي الفتحة (نقرتان: أعلى وأسفل السحاب) → الأداة تتتبع edge loops الحافتين.
- سلسلتا عظام على الحافتين (يمين/يسار الفتحة).
- **عظمة Slider** تنزلق على منحنى وسطي (Follow Path، offset مربوط بخاصية 0–1).
- درايفرز: كل زوج عظام يتباعد بمقدار يعتمد على (موضعه − موضع الـ slider) — فوق الـ slider مغلق، تحته مفتوح ويتدلى للجانبين.
- بديل أبسط كمرحلة أولى: **شيب كي واحد** (مغلق→مفتوح) يقوده الـ slider — أسهل بكثير لكن أقل مرونة. اقتراحي: نبدأ بالشيب كي ونرقّي لاحقاً.

### 3.6 وحدة المقتنيات (Props)
**تغطي:** حقيبة ظهر، حقيبة يد نسائية، سيف، عصا، درع، هاتف، كوب — أي غرض تمسكه أو تحمله الشخصية.

**التصميم:**
- عظمة Prop رئيسية + **Parent Switching** جاهز: عالم / يد.R / يد.L / ظهر / ورك — التبديل بسلايدر مع حفاظ على الموضع (بدون قفزة). السيف ينتقل من الغمد لليد بحركة واحدة.
- **حقيبة الظهر:** Rigid على الظهر + الأحزمة بوحدة الحبل (3.2) + اهتزاز خفيف اختياري.
- **حقيبة اليد:** Rigid + المقبض سلسلة قصيرة متدلية (وحدة 3.2) — تتأرجح مع المشي.
- **السيف/العصا:** عظمة Prop + socket غمد/ظهر + IK لليد الثانية للمسك بيدين (العصا).
- Snap تلقائي: زر "In Hand" يطابق الـ Prop مع عظمة اليد بمحاذاة قابلة للضبط ثم يحفظها offset.

## 4. الكشف الذكي (Smart Detection)

"الذكاء" = تعرّف هندسي قوي (نفس فلسفة الجسم — بدون شبكات عصبية، بصراحة مع المستخدم).

**الخوارزمية:** عند اختيار ميش ملابس والجسم مُريَّق:
1. أي عظام جسم تقع داخل الحجم المحيط بالقطعة؟
2. الشكل الطوبولوجي: أسطوانة مفتوحة؟ شريط؟ صفيحة مغلقة؟
3. القرار:

| السمة الهندسية | الاستنتاج |
|----------------|-----------|
| يحيط بالفخذين/الساقين، أسطوانة مفتوحة | تنورة/كندورة (الطول يحدد الـ Preset) |
| شريط رفيع يمر بالرقبة ويتدلى | ربطة/Lanyard |
| يغلف الجذع + الذراعين | قميص/جاكيت → Fitted |
| فوق عظمة الرأس، حجم صغير | قبعة → Rigid |
| يحيط بالقدم | حذاء → Rigid+toe |

4. **الأداة تقترح والمستخدم يؤكد** (dropdown قابل للتعديل) — لا نجزم أبداً، والتأكيد نقرة واحدة.

## 5. نظام الوجه المعياري (Face System)

**الأساس:** كورس Storm (موجود كاملاً كـ skill: facial-rigging-blender) — ribbon الشفاه، lip zipper، auto-blink، eyelids follow، حواجب/خدود بشيب كيز + Geometry Nodes weight splitting، correctives لزوايا الفم.

**فكرتك المعيارية — التصميم:**
كل منطقة وجه = slot، وكل slot له أكثر من **وصفة (Recipe)**:

| المنطقة | Recipe A (افتراضي) | Recipe B | Recipe C |
|---------|--------------------|----------|----------|
| الشفاه/الفم | Storm ribbon + zipper | Bone-based بسيط (نمط ARP) | Shape-key فقط (نمط FaceIt) |
| العيون | Storm (auto-blink + follow) | Bone-based (نمط ARP) | Shape-key (نمط FaceIt) |
| الحواجب | Storm (shape keys + GN split) | Bones مباشرة | — |
| الخدود | Storm | — | — |
| الفك/اللسان/الأسنان | Storm | — | — |

- ماركرز وجه مشتركة (زوايا العين، خط الشفاه، الحاجب...) تخدم كل الوصفات — المستخدم يضعها مرة واحدة ويبدّل الوصفات كما يشاء.
- عندنا جاهز من الـ skill: سكربت mirror sub-targets + سكربت auto driver assignment.
- **التنفيذ تدريجي:** slot الفم بوصفة Storm أولاً (الأصعب والأعلى قيمة)، ثم العيون، ثم الباقي. الوصفات البديلة (B/C) بعد اكتمال Storm الكامل.

## 6. Deform Toolkit (شيب كيز / درايفرز / كوركتيف)

أدوات مستقلة تعمل مع أي ريق (حتى غير SmartRig) — قيمة تسويقية بحد ذاتها:

1. **Driver Wizard:** حدد شيب كي + عظمة → درايفر جاهز (المسافة/الزاوية/المحور من قائمة). النواة موجودة في سكربت skill الوجه.
2. **Corrective Shape Key Wizard:** خذ البوز المشكِل → زر "Sculpt Corrective" (يفتح Sculpt على نسخة) → زر "Finish" ينشئ الشيب كي بالفرق + درايفر تلقائي على زاوية العظمة (Rotational Difference). هذا يختصر أكثر workflow مؤلم في الريقينغ إلى زرين.
3. **Corrective Bone Wizard:** نفس الفكرة بعظمة: عظمة مساعدة + Transform Constraint من زاوية المفصل (مرفق/ركبة/كتف) — Presets جاهزة للمفاصل الشائعة.
4. **Shape Key Manager:** قائمة مجمعة، mirror، انسخ من ميش لميش (للملابس: نقل correctives الجسم للقميص).

## 7. نظام الأنيميشن (تبويب ANIMATE)

الهدف: المستخدم لا يبدأ من الصفر — حلول ذكية فوق الريق، كلها **أدوات بلندر أصلية** (درايفرز، Action Constraints، قيود) = تعمل في الرندر بدون سكربتات وقت التشغيل، وصالحة لإنتاج سينمائي.

### 7.1 جهاز القيادة (Locomotion Controller) — الميزة الرائدة
عظمة قيادة واحدة تحرك الشخصية بالكامل:

| حركة عظمة القيادة | النتيجة |
|--------------------|---------|
| سحب للأمام/الخلف | الشخصية تمشي (دورة مشي تلقائية بطور مربوط بالمسافة) |
| زيادة السرعة | مزج تلقائي مشي → ركض |
| دوران | الشخصية تدور مع دوران الجسم الطبيعي |
| التوقف | مزج إلى وقفة idle |
| إنزال | وقوف → قرفصاء → جلوس → رقود (عتبات ارتفاع) |
| رفع فوق الوقوف | نهوض / قفز |

**الآلية التقنية:**
1. **مكتبة أكشنات مدمجة** (مشي، ركض، idle، جلوس، رقود، قفز، نهوض) نصنعها مرة واحدة على ريق SmartRig — ممكنة لأن تسمية العظام تسميتنا وثابتة.
2. طور الدورة = المسافة المقطوعة ÷ طول الخطوة (درايفر على خاصية phase).
3. **Action Constraints** على عظام التحكم تقيّم الأكشن عند الطور المحسوب.
4. المزج بين الحالات (مشي/ركض/جلوس...) = مزج تأثير الـ Action Constraints عبر درايفرز على السرعة والارتفاع.
5. **زر Bake:** تحويل النتيجة إلى كي-فريمز عادية للتلميع اليدوي — أساسي للإنتاج السينمائي، الجهاز يعطي blocking جاهز والأنيماتور يصقل فوقه.

### 7.1.1 التعديل فوق الأنيميشن التلقائي (مبدأ غير قابل للتفاوض)
المستخدم يجب أن يعدّل ويضيف أنيميشن خاصاً به فوق ناتج الجهاز، بمستويين:

- **مستوى حي (Additive Layer):** الـ Action Constraints تُركَّب على عظام **MCH وسيطة** وليس على عظام التحكم مباشرة. عظام التحكم المرئية تتبع الـ MCH لكنها تبقى **حرة تماماً** — أي تحريك يدوي لليد أو الرأس أو العمود يُضاف فوق دورة المشي وهي شغالة. مثال: الشخصية تمشي بالجهاز والمستخدم يحرك يدها تلوّح — بدون Bake، بدون كسر أي شيء.
- **مستوى Bake:** زر يحوّل كل شيء (الجهاز + التعديلات الحية) إلى كي-فريمز عادية على عظام التحكم، ثم يعطّل الجهاز — سيطرة كاملة 100% بالطريقة التقليدية، والـ NLA يبقى متاحاً للطبقات.
- سلايدر **Influence** عام للجهاز: من 0 (يدوي كامل) إلى 1 (جهاز كامل) — والقيم الوسطى تمزج، لكل جزء جسم على حدة (سفلي/علوي/رأس).

### 7.2 باكات الحركة (Action Packs)
نفس آلية 7.1 — أكشنات جاهزة تُستدعى من الـ UI:
- **باك القتال:** لكمة، ركلة، تفادي، حراسة — أزرار/سلايدرات تشغيل مع مزج.
- **أنماط المشي (Presets):** طبيعي، كرتوني، تسلل، عسكري، أنثوي — تستبدل أكشن المشي في جهاز القيادة.
- البنية مفتوحة: باكات إضافية مستقبلاً (رقص، حيوانات...) بدون تعديل الكود.

### 7.3 الصوت → حركة الفم (Lipsync)
مستويان بصراحة:
- **بسيط (مدمج، بدون تبعيات):** تحليل amplitude للملف الصوتي (normalize ثم envelope) → كي-فريمز على فتح الفك + شيب كي الفم. جيد للخلفيات واللقطات البعيدة.
- **احترافي:** تكامل مع **Rhubarb Lip Sync** (مجاني، مفتوح المصدر، معيار صناعي) — يستخرج الفونيمات من الصوت، ونحن نربط كل فونيم بشكل فم (viseme) من مكتبة البوزات. المستخدم يعطي ملف wav → أنيميشن فم كامل.
- يعتمد على اكتمال Face System (أشكال الفم) — لذلك ترتيبه بعد الوجه.

### 7.4 مكتبة البوزات والتعابير
- **Pose Assets** أصلية في بلندر، تُشحن مع الإضافة: تعابير وجه كاملة (فرح، غضب، حزن، دهشة...) + أشكال الفم (visemes) + بوزات أيدٍ شائعة.
- هي نفسها مصدر أشكال الفم للـ Lipsync — بنية واحدة تخدم غرضين.

### 7.4.1 مصادر الأكشنات واستراتيجية الترخيص (قرار مهم)
- **iClone 8 / CC4 + باكات الأنيميشن (مشتراة):** EULA ريلوجن يمنع إعادة توزيع المحتوى (حتى المُصدَّر FBX) داخل منتج طرف ثالث — التصدير للاستخدام الخارجي يتطلب Standard License، وإعادة البيع/التوزيع كأصول قابلة لإعادة الاستخدام ممنوعة صراحة. **الاستخدام الآمن:** أفلام سعيد الخاصة + مرجع ونماذج تجريبية أثناء التطوير فقط.
- **ما يُشحن مع الإضافة:** أكشنات **أصلية نصنعها بأنفسنا** على ريق SmartRig (نستخدم باكات iClone كمرجع بصري للجودة، لا ننسخ البيانات).
- **الميزة القاتلة بدلاً من شحن مكتبات الغير: Universal Retargeter** — زر "Import Animation" يستورد FBX من iClone/CC4/Mixamo/ActorCore ويعيد استهدافه تلقائياً على ريق SmartRig (خريطة أسماء عظام جاهزة لكل مصدر). كل مستخدم يستعمل باكاته المشتراة قانونياً + يفتح الإضافة على كل مكتبات الأنيميشن في العالم.

### 7.3.1 قرار تقنية الـ Lipsync (محدَّث)
- **Rhubarb Lip Sync:** رخصة MIT، ملفات تنفيذية جاهزة لويندوز وماك ولينوكس — **يُشحن مدمجاً مع الإضافة قانونياً** (مع ملف الرخصة). هذا هو المسار الافتراضي لكل المنصات.
- **NVIDIA Audio2Face:** أصبح مفتوح المصدر (SDK برخصة MIT + الموديلات على Hugging Face) — جودة أعلى بكثير (عواطف + وجه كامل)، لكنه يدعم **ويندوز ولينوكس فقط** (يفضّل GPU إنفيديا، مع CPU fallback) — **لا يدعم ماك**. القرار: مسار اختياري premium بزر "Download" داخل الإضافة (الموديلات كبيرة الحجم)، وليس بديلاً عن Rhubarb.

### 7.6 طبقات الأنيميشن (Animation Layers)
NLA بلندر قوي لكن غير ودود — نبني فوقه واجهة طبقات بسيطة (نمط برامج الأنيميشن الاحترافية):
- قائمة طبقات: إضافة / كتم / solo / دمج / حذف — كل طبقة = NLA strip بوضع Additive تُدار تلقائياً.
- تكامل مع جهاز القيادة: ناتج الجهاز = الطبقة الأساسية، والمستخدم يضيف طبقات تعديل فوقها (تلويح يد، حركة رأس) بدون لمس الأساس.
- تكامل مع باكات الحركة: اللكمة تُسقط كطبقة فوق المشي وتُمزج تلقائياً.
- زر Bake النهائي يدمج الطبقات كلها في أكشن واحد.

### 7.5 التكيف مع الأرضية (Ground Adaptation)
- المستخدم يحدد ميش الأرضية → أهداف IK للقدمين تحصل على **Floor Constraint + Shrinkwrap (project)** ضدها.
- القدم تلتصق بالأرضية غير المستوية + تعويض ميلان القدم حسب normal السطح.
- مدمج مع جهاز القيادة: الشخصية تمشي على تضاريس (درج، تلة) تلقائياً.
- حدود صريحة: لا يحل محل تصحيح يدوي في اللقطات القريبة المتطرفة — لكنه يوفر 90% من العمل.

## 7.7 تصميم الواجهة (UI/UX)

**المبدأ الحاكم:** الواجهة تعرض "الخطوة القادمة" وليس كل الأدوات. مبتدئ = أزرار كبيرة و Presets؛ محترف = نفس الواجهة + أقسام Advanced قابلة للطي. لا واجهتين منفصلتين — واجهة واحدة بمستويي كشف (Progressive Disclosure).

### أ) تبويب SmartRig في السايدبار (N-Panel)

```
┌─ Soulify ────────────────────┐
│ [شخصية: Storm ▾]   [⚙ Simple|Pro] │
├───────────────────────────────────┤
│ ● RIG                              │
│  ✔ 1. الجسم        [تعديل]        │
│  ✔ 2. الوجه        [تعديل]        │
│  ▶ 3. الملابس     [+ إضافة قطعة]  │
│     👕 قميص      ✔                │
│     👗 تنورة     ⏳ (خطوة 2/4)    │
│  ▶ 4. المقتنيات   [+ إضافة]       │
├───────────────────────────────────┤
│ ● ANIMATE   (يُفتح بعد الريق)     │
│  🚶 جهاز القيادة                  │
│  🥊 باكات الحركة                  │
│  🎤 Lipsync                       │
│  😀 مكتبة البوزات                 │
│  ⛰ الأرضية                        │
├───────────────────────────────────┤
│ ● TOOLS (Deform Toolkit)          │
└───────────────────────────────────┘
```

- **كل خطوة صف واحد:** أيقونة + اسم + حالة (✔ مكتمل / ⏳ جارٍ / ⚠ مشكلة) + زر إجراء واحد. النقر يوسّع بارامترات الخطوة.
- **إضافة قطعة ملابس:** زر واحد → الإضافة تحلل الميش المحدد وتقترح النوع (Smart Detection) → المستخدم يؤكد من شبكة أيقونات (تنورة/كندورة/ربطة/قبعة/حذاء/سحاب...) → Preset + زر Build.
- **نظام التحقق المدمج:** الأفخاخ الموثقة من كورس Storm (Deform checkbox، تطابق الأسماء، الأوزان...) تظهر كصف ⚠ مع زر **"إصلاح"** بجانبها — ليس رسالة خطأ بل حل بنقرة.
- **Simple/Pro:** مفتاح في رأس اللوحة. Simple يخفي: الأسماء التقنية، أوزان يدوية، بارامترات دقيقة. Pro يظهر كل شيء + subpanel "Advanced" لكل وحدة. يُحفظ في Preferences.
- **زر ? لكل قسم** يفتح شرحاً قصيراً (tooltip موسع أو رابط).

### ب) لوحة Item (السياقية — الأهم للأنيماتور)

مثل Rigify/CloudRig: عند تحديد عظمة، تظهر خصائصها **هي فقط** في N > Item:

| العظمة المحددة | ما يظهر |
|----------------|---------|
| سلايدر السحاب | Zipper 0–1 |
| كنترول الشفاه | Lip Zip L/R، Squash |
| عظمة العين | Auto Blink، Blink Height، Eyelid Follow، Space (Head/Body/World) |
| جذر التنورة/الكندورة | Follow Legs، No-Intersect، Stiffness |
| عظمة Prop (سيف/حقيبة) | Parent: Hand.R/Hand.L/Back/World + Snap |
| عظمة القيادة | Influence لكل جزء جسم، نمط المشي، Bake |

التنفيذ: custom properties على العظام + metadata للوحدة، واللوحة تقرأ الوحدة المالكة للعظمة المحددة. **الأنيماتور لا يفتح تبويب الإضافة أصلاً أثناء العمل** — كل شيء في Item.

### ج) إضافات UX

- **Pie Menu** (زر اختصار): IK/FK snap، Space switch، مكتبة البوزات، بوز → عكس — للأنيماتور السريع.
- **مكتبة البوزات عبر Asset Shelf** الأصلي (مثل Storm) — معاينات مصغرة جاهزة.
- **شبكة الـPresets بمعاينات مصورة** (أيقونة لكل نوع قطعة/نمط مشي) بدل قوائم نصية.

### د) الأيقونات المخصصة (سعيد يزوّدها)

- التقنية: `bpy.utils.previews` — **PNG 32×32** (يفضل معها نسخة 64×64). SVG غير مدعوم مباشرة في بلندر.
- النمط المقترح: أحادية اللون بخلفية شفافة (تتكيف مع ثيمات بلندر الفاتحة/الداكنة)، خط ثابت السماكة.
- القائمة المطلوبة (المرحلة الأولى ~24 أيقونة): body, face, skirt, kandura, dress, tie, lanyard, hat, shoe, sandal, shirt, pants, jacket, zipper, backpack, handbag, sword, staff, drive, fight, lipsync, poses, ground, layers + حالات (check, warning, build).
- تسمية موحدة: `srp_<name>.png`.

## 8. مخاطر وملاحظات صراحة

- **عدم التداخل** لن يكون مثالياً 100% في البوزات المتطرفة — Shrinkwap حل تقريبي ممتاز عملياً، ونوثق حدوده.
- **Attach to Body** جودته مرهونة بجودة أوزان الجسم نفسه — أي تحسين بأوزان الجسم ينعكس على كل الملابس.
- **السحاب** يحتاج نموذج تجريبي قبل وعد المستخدمين به.
- **قواعد الـ skill سارية:** أي تعديل = مصدر → bump نسخة → zip → إعادة تثبيت؛ التحقق رقمياً لا بالعين؛ rest coords للملابس (الدرس مُوثق).
- كل وحدة تُختبر على ميش حقيقي قبل الانتقال للتالية — لا نبني الخمس دفعة واحدة.

## 10. ملاحظة معمارية للأنيميشن

كل ميزات تبويب ANIMATE مبنية على 3 لبنات أصلية فقط: **Action Constraints + درايفرز + قيود** (Floor/Shrinkwrap/Child-Of). لا سكربتات وقت تشغيل، لا handlers دائمة = الملف يفتح عند أي مستخدم بدون الإضافة مثبتة (الريق يعمل standalone)، ويرندر على مزارع الرندر بدون مشاكل. هذا شرط أساسي للإنتاج السينمائي ويجب ألا نكسره في أي ميزة.
